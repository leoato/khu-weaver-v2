## 2. 학번별 교육과정 추가
- 전자공학부 홈페이지에 2020~2024 교육과정 PDF가 모두 존재(크롤링 경로 확보됨).
- **선행조건:** 아래 **3-1 데이터 정리법(스키마/파이프라인)** 을 먼저 확립한 뒤 진행.
- 작업 단위: 학번 1개 = `js/data/EE_<연도>.js` 파일 1개 + `registry.js`의 `curriculumYears`에 연도 추가.

## 3. 학과별 확장
- 현재 전자공학과(EE)만 활성. 생체의공학과(BME)·컴퓨터공학과(CSE) 등은 "준비 중" 상태.
- **3-1. (최우선) 데이터 정리법 확립 — 분업의 핵심:**
  - 2·3번 진행 전에 **표준 데이터 스키마 + 작성 가이드 + 검증 스크립트**를 먼저 강하게 세팅.
  - 목표: 팀원 누구나 같은 포맷으로 데이터를 만들어 넣을 수 있게 → 분업 효율 극대화.
  - 포함할 것:
    - `courses` / `prerequisites` / `tracks` 필드 표준 정의서 (필드명·타입·허용값 — 예: track 코드 MATH/PHYS/COMP/CS/CSY/SW)
    - 파일 네이밍 규칙: `<학과코드>_<학번>.js` (예: `EE_2026.js`)
    - 신규 데이터 추가 시 체크리스트 (registry 등록, index.html 로드, 무결성 검증)
    - 무결성 검증 스크립트 상시화 (과목코드 중복/선수과목 참조 유효성/학점 합계/트랙 코드 유효성 — v1.5에서 쓴 검증 로직을 재사용 가능한 형태로)
  - 산출물 예시: `docs/DATA_SCHEMA.md`, `tools/validate_curriculum.js`
- **3-2. 학과 분류 체계는 학교 공식 분류를 따른다:**
  - 이유: 크롤링 시 학교 사이트 구조와 1:1로 매칭되어 수집이 편함.
  - 학과코드도 가능하면 학교/홈페이지에서 쓰는 식별자에 맞춰 정함.

## 4. 성적 입력 간소화 — LLM 캡처 변환 플로우
- **문제의식:** 사용자가 수업별 학점을 일일이 손입력하는 건 번거로움.
- **아이디어:** 에브리타임/인포21(경희대 학력포털)에서 성적을 **캡처 → LLM(GPT/Gemini/Claude)** 에 이미지+프롬프트 입력 → KHU-Weaver에 바로 붙여넣을 수 있는 **데이터셋(JSON 등)** 으로 변환.
- **구성요소:**
  - 4-1. 변환용 표준 프롬프트 제공 (사용자가 복붙해서 쓸 수 있게). 출력 포맷 = KHU-Weaver 연동 스키마.
  - 4-2. 변환 결과를 사이트에 붙여넣으면 자동 연동되는 입력창.
  - 4-3. 사이트에 사용법 설명 + **애니메이션 가이드**(캡처→LLM→붙여넣기 단계를 시각적으로 안내)로 이해도 ↑.
- 고려: 캡처 이미지 안의 개인정보(이름/학번 등) 처리 안내 문구 포함.

## 5. 성적 계산기 UX 수정 (계절학기/학기 탭)
- **5-1. 계절학기 구분 모호:** 현재 "계절"이 **몇 학년의 여름/겨울인지 구분 불가**(예: 2학년 여름 vs 1학년 겨울). → 계절학기에 **연도·계절(여름/겨울) 메타** 부여 또는 학기 라벨 체계 재설계.
- **5-2. 4학년 2학기 탭 가려짐:** 마지막 학기 탭이 화면에 가려져 키보드 방향키로만 접근 가능 → **직관적이지 않음**. 탭 가로 스크롤/줄바꿈/반응형 처리 필요.

## 7. 선배 데이터 수집 방법 (핵심 가치 — 기획 필요)
- 이 사이트의 의미는 **선배들의 실제 이수/진로 정보**에서 나옴. 수집 방법·항목 고민 필요.
- **수집하고 싶은 정보:**
  - 어떤 수업을 들었는지(이수 과목 빌드)
  - 어느 분야로 특화되었는지(트랙)
  - 어느 회사/대학원에 갔는지(진로 결과)
- **수집 경로 후보:** 선배 에타 성적/시간표 공유 요청 등 — 방법론·개인정보 동의 절차는 추후 설계.
