## 1. 아키텍처: 왜 이 구조인가

```
[페이블 세션(본부)]                [소넷 세션들(작업반)]
 스키마·검증기·브리프 제작   →     브리프 파일 읽고 실행만
 결과 검토·통합 승인         ←     STATUS.md에 결과 보고
```

- 소넷 세션은 **코드베이스 전체를 읽을 필요 없음** — 브리프 파일 1개 + 스키마 1개 + 참고 데이터 파일 1개만 읽으면 충분하도록 브리프를 자기완결적으로 작성함 (토큰 절약의 핵심)
- 검증기가 품질 게이트 역할 → 소넷 결과물의 품질 편차를 기계적으로 걸러냄
- 병렬 가능: 크롤링 세션들은 서로 다른 파일을 만들므로 충돌 없음

## 4. 타과 확장에 대한 답 (사용자 질문: "그냥 적용만 시키면 되잖아?")

**80%는 맞고 20%는 아님.** 재사용되는 것: 스키마, 검증기, 브리프 템플릿, registry 구조, 크롤링 절차.
학과마다 새로 해야 하는 것 2가지:
1. **트랙 체계 정의** — [별표2] 전공심화 분류가 학과마다 다름 → track 코드·tracks 키·레이더 축을 학과별로 정의하고 app.js 레이더 로직 일반화 필요 (현재 EE 5트랙 하드코딩)
2. **크롤링 경로 확보** — 학과 홈페이지 구조가 다름 → 학과당 1회 페이블/사용자가 PDF 경로 확인
즉 "학과 1개 추가 = 소넷 크롤링 세션 N개 + 페이블 트랙정의 1회"로 공식화됨. 공모전 문서에는 이 확장 공식 자체를 어필 포인트로 쓸 것.
