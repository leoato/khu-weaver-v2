# [브리프] U-2: 에타 스크린샷 → Gemini OCR 자동 입력 (JS 수정 허용)

> 대상 모델: 오퍼스 권장 · 발행: 페이블 2026-07-05 · **선행조건: U-1(온보딩 개편) 완료 후**
> 작업 폴더: `khu-weaver_v2/` 만. 설계 기준: `khu-weaver_v2/docs/UX_FLOW.md`의 "A안 세부"

## 아키텍처 (사용자 확정 사항)
- **프로토타입 단계: API 키를 클라이언트에 직접 삽입.** 사용자에게 키 발급을 시키지 않는다.
- 키는 `js/config.js` 신설: `window.KW_CONFIG = { GEMINI_API_KEY: "여기에_키" };`
  - 파일 상단 주석: "⚠️ 프로토타입 전용. 공개 배포 전 반드시 프록시 서버로 이전(키 유출 위험)."
  - index.html에서 다른 스크립트보다 먼저 로드. 키가 빈 문자열이면 A카드에 "준비 중" 배지 표시하고 B안으로 폴백.
- 호출: `POST https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=...`
  이미지 inline_data(base64) + `generationConfig.responseMimeType: "application/json"`.
  ※ 모델명은 구현 시점에 사용 가능 여부 확인, 불가하면 최신 flash 계열로 대체하고 STATUS에 기록.

## 기능 스펙
1. **업로드**: 1~8장(jpg/png), 드래그앤드롭+버튼, 8장 초과 시 안내. 전송 전 캔버스 리사이즈(긴 변 1536px)로 용량·토큰 절약.
2. **프롬프트**: "경희대 에브리타임/인포21 성적 스크린샷이다. 과목명, 학수번호(있으면), 학점(credits), 성적(grade: A+/A0/B+... 문자), 수강학기(있으면)를 JSON 배열로만 응답" + 출력 스키마 예시 포함. 여러 장은 한 요청에 다중 이미지로.
3. **매칭(전공만 — U-3 결정)**: 응답 과목 중 **현재 프로필 학번의 CURRICULA에 있는 전공 과목만** 대상으로 대조. 교양 등 미등재 과목은 결과표에 "전공 외 — 제외됨" 회색 섹션으로만 표시(적용 안 함). 성적은 과목 귀속 구조(TASK_GPA_MAJOR 참조)로 저장. 매칭 순서: — ①학수번호 정확일치 ②과목명 정규화(공백·특수문자 제거) 일치 ③실패 항목은 결과표에 노랑 표시+드롭다운으로 수동 매칭.
4. **결과 확인표**: 인식 과목·성적 목록, 체크박스로 제외 가능, [적용] 시 이수체크+GPA 계산기에 반영(기존 localStorage 계약 준수, app.js 계산 로직 수정 금지 — 데이터 주입만).
5. **직조 애니메이션**: 분석 중 Weaver 컨셉 — 씨실·날실이 교차하며 짜이는 CSS 애니메이션 + 문구 순환("시간표를 읽는 중…", "과목을 대조하는 중…", "빌드를 짜는 중…") + `n/8장` 진행 표시. 외부 라이브러리 금지, 순수 CSS/JS.
6. **개인정보 안내 1줄** + 오류 시(네트워크/파싱 실패) 좌절 없는 폴백: "직접 선택하기로 이어서 할 수 있어요".

## 검증 (필수)
1. node --check 전 JS + 로드 시뮬레이션
2. API 목업 테스트: fetch를 스텁으로 갈아끼운 테스트 HTML(tests/ 임시 폴더)로 매칭·확인표·적용 흐름 검증 — 실키 없이도 로직 검증 가능해야 함
3. 실키 테스트는 사용자에게 요청(키 삽입은 사용자가 직접)
4. STATUS.md 1줄 보고 (| U-2 | ... |)
