# API 키 보안 메모

## 결론

브라우저에 내려가는 파일에는 API 키를 넣으면 안 됩니다. `config.js`, HTML, 클라이언트 JavaScript에 들어간 값은 사용자가 개발자 도구나 네트워크 기록으로 확인할 수 있습니다.

이 프로젝트는 배포 시 다음 구조를 사용합니다.

1. 브라우저는 `/api/ocr`로 이미지 분석 요청을 보냅니다.
2. Netlify Function이 서버에서 `GEMINI_API_KEY` 환경변수를 읽습니다.
3. Netlify Function이 Gemini API를 호출하고 결과만 브라우저로 돌려줍니다.

## 이미 키를 하드코딩한 적이 있다면

이미 GitHub, 캡처 화면, 배포 사이트, 공유 압축파일 등에 실제 키가 들어간 적이 있다면 그 키는 노출된 것으로 보는 편이 안전합니다.

권장 조치:

- 기존 Gemini API 키를 폐기하거나 제한합니다.
- 새 API 키를 발급합니다.
- 새 키는 Netlify 환경변수 `GEMINI_API_KEY`에만 저장합니다.
- `js/config.js`의 `GEMINI_API_KEY`는 빈 문자열로 유지합니다.

## 추가 비용 보호

현재 함수는 같은 배포 도메인에서 온 브라우저 요청만 허용하도록 출처를 확인합니다. 배포 환경에서는 `Origin` 헤더가 없는 직접 호출도 차단하고, 로컬 개발 주소만 예외로 둡니다.

또한 함수는 OCR 요청 형태만 받도록 제한합니다. 텍스트 프롬프트와 JPG/PNG 이미지 1~8장이 포함된 요청만 Gemini로 전달하고, 그 외 요청은 Gemini 호출 전에 거절합니다. 브라우저에서는 최대 20장까지 받을 수 있지만, 서버에는 8장 이하 묶음으로 나눠 전송합니다.

사용량 초과(429)나 일시적인 Gemini 오류가 발생하면 Netlify Function은 `GEMINI_FALLBACK_MODELS`에 설정된 보조 모델로 한 번 더 시도합니다. 모든 모델이 사용량 제한에 걸리면 사용자에게 잠시 후 재시도하거나 직접 선택으로 이어가도록 안내합니다.

다만 이 방식은 공개 체험용 기본 보호입니다. API 사용량 제한은 Google AI Studio 또는 Google Cloud 쪽에서도 설정하는 것이 좋습니다.
